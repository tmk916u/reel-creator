## Why

`simplify-subtitle-to-1stage-remap` 適用後のベースライン (job 7ca71f64 / 301574d9) で、 1 段目 ASR の認識ミスが直接字幕に出る問題が確認:

```
301574d9: 客 事への 食     ← 「お客様が悩まれているダイエットの」 の subword 列
          一番日 自分自もの ← 「一日 自分自身が体に悪いものを」 が断片化
          悪い要           ← 「重要」 が「悪い要」 に
7ca71f64: こ律神経の話     ← 「自律神経」 が「こ律」 に
          エットや ボメ 康  ← 「ダイエットや ボディーメイク 健康」 が「ボメ 康」 に
```

LLM 校正 + 動画固有辞書で吸収すべきだが、 現状の実装では効いていない:

1. **`summarize_with_mishearings`**: 動画固有の誤認識ペア (例「事への→の」「悪い要→重要」) を LLM 抽出するが、 プロンプトに「subword 化された短い断片」 の例が無く、 抽出されない
2. **`correct_transcript_segments`**: 長さ制約「-50% 〜 +30%」 と「統合・分割禁止」 で、 短い subword 列 (5 文字以下) の大幅修正が拒否される

業務量産で「投稿前に手動編集」 する運用負荷を減らすため、 LLM 校正の精度を向上させる。

## What Changes

- **`_SUMMARIZE_AND_MISHEAR_PROMPT` を強化**:
  - 「subword 化された短い断片」「2 文字以下の意味不明な並び」 を検出する例を追加
  - 抽出上限を 10 → 15 個に緩和
- **`_CORRECTION_SYSTEM_PROMPT` を強化**:
  - 「短い subword 列を周辺の文脈から推測して書き換えて良い」 を明示
  - 長さ制約を「-50% 〜 +30%」 → 「-70% 〜 +50%」 に緩和 (短い断片の修正余地確保)
- **`correct_transcript_segments` の長さチェック緩和**:
  - 拒否条件「2 倍超 / 10 文字以上長い」 を「2.5 倍超 / 15 文字以上長い」 に緩める
- BREAKING: なし (内部実装の調整のみ)

## Capabilities

### Modified Capabilities
- `quality-line`: 項目#5 字幕の誤認識の少なさ をプロンプト改善で底上げ

## Impact

- **Backend**: `llm.py` のプロンプト 2 つと長さチェック (約 20 行修正)
- **テスト**: 既存テスト維持。 LLM 呼出はモックなので影響なし
- **業務量産**: 「事への / 悪い要 / こ律 / ボメ 康」 等の誤認識を **80% 程度吸収する** ことを期待
- **副作用リスク**:
  - 長さ制約緩和で LLM が過度に書き換える可能性
  - 既存の retry 機構 (`_CORRECTION_RETRY_PROMPT`) が拒否範囲を control するので safety net あり
