## Context

### 直近の動向
過去 30 コミットで、ユーザーの体感フィードバックを起点として **施策A-G** + **動画文脈サマリー** + **動画固有辞書** + **1段目primary戦略** などを実装してきた。これらは確実に品質を上げているが:

- 「どこまで実装すれば業務量産投入可能か」 が暗黙のままだった
- フィードバックのたびに 1 つ修正 → 別の場所が壊れる、 を繰り返す
- ベースラインがないので、 退行を機械的に検知できない
- どの動画ジャンル/サブトピックで品質がブレるか把握できていない

### この change のゴール
**実装の改善ではなく、 「完成定義の確定 + 現状の測定 + 不合格項目の可視化」**。これにより:
1. 業務量産投入の可否を **定量判定** できる
2. 次に何を直すべきかが明確になる
3. 修正後の退行をチェックできる(回帰テスト)

### 既存資産
- `openspec/changes/add-ai-jump-cut/` (37/38 完了、 同等の領域)
- `openspec/specs/` は空(本 change で初の永続 spec を作成)
- `backend/tests/` ユニットテストは 105 件 PASS、 ただし**動画レベルの統合テストはゼロ**

## Goals / Non-Goals

**Goals:**
- 業務量産投入合格の **14 項目チェックリスト** を永続 spec として確立
- 業務代表動画 **3 本** を `test-videos/` に整備し、 各動画に期待値ドキュメントを付与
- 14 項目のうち **機械的判定可能なもの** を測定する Python スクリプトを提供
- 目視判定項目には Markdown チェックリストテンプレを提供
- **現状のベースライン**を 1 度測定し、`baseline.md` に記録
- 不合格項目を **後続 change** として枝分かれする運用ルールを確立

**Non-Goals:**
- パイプラインのコード改善(別 change で扱う)
- フロントエンド UI の変更
- 新しい AI 機能の追加
- 完全自動の品質評価 AI(Claude Vision など)— これは長期 backlog
- 5 本以上のテスト動画拡張 — まず 3 本で。

## Decisions

### D1: チェックリストの粒度を 14 項目に絞る
**なぜ 14 項目か**:
- 多すぎると測定コストが高く、 ユーザーの「全部やる」モチベを削ぐ
- 少なすぎると業務量産品質の網羅性に欠ける
- 4 カテゴリ × 平均 3-4 項目 = 14 項目で「現実的に毎週測定可能」な粒度

**Alternatives 検討**:
- 全網羅(30+ 項目): 測定が重く運用に乗らない → ❌
- 5 項目程度: 細部の品質が見落とされる → ❌

### D2: spec ファイル配置を `openspec/specs/quality-line/spec.md` 単独に
**なぜ単独 capability か**:
- 14 項目は相互依存する一体の品質基準で、 別 capability に分けると参照が複雑化
- 後続の改善 change から **単一の参照点** にできる

### D3: 機械測定と目視判定を分離
| 項目 | 判定 |
|---|---|
| #4 動画長 / #14 処理時間 | 機械(数値しきい値) |
| #6 タイミング同期 / #8 文字数分布 | 機械(統計) |
| #7 助詞直後 flush | 機械(末尾文字パターン照合) |
| #11 CTA 豆腐 | 機械(フォントカバレッジチェック) |
| #1 ストーリー保全 / #5 誤認識 / #9 HOOK | 目視 |
| #10 テロップ被写体保護 | 目視(スクリーンショット確認) |

**なぜ完全自動化しないか**:
- 動画レビュー AI(Claude Vision) は API コスト高 + 不確実性高
- 目視チェックは整体院の業務担当者(ユーザー)が **1 動画 5 分以内** で完了できる粒度に保つ

**Alternatives**:
- Claude Vision で全項目自動: コスト高、 後で検討 → 長期 backlog
- Snapshot Test(出力字幕/HOOK/CTA を文字列で固定): LLM 出力のブレで毎回失敗 → ❌

### D4: テスト動画 3 本の選定基準
| ファイル名 | 想定特性 |
|---|---|
| `seitai_standard.mov` | 整体・血流・筋肉系の "標準" 動画(2-3 分) |
| `seitai_food.mov` | 整体師が食事・ダイエットを話す "サブトピック" 動画 |
| `seitai_long.mov` | 5 分以上の長尺、 言い直し・間が多めの "ストレステスト" |

**なぜ 3 本か**:
- 1 本: ばらつき検出不可
- 3 本: カテゴリ別の代表+ストレステスト網羅
- 5 本以上: 整備コストが高い、 まず 3 本で

### D5: ベースラインの記録形式
`openspec/changes/establish-quality-line/baseline.md` に Markdown テーブルで:

```
| 項目# | 項目名 | seitai_standard | seitai_food | seitai_long | 合格率 |
|------|--------|-----------------|-------------|-------------|--------|
| 1    | ストーリー保全 | ✅ | ❌ | ⚠️ | 1/3 |
| ...  | ...    | ... | ... | ... | ... |
```

不合格項目には **コメント欄** で具体的な状況を記録し、 後続 change の入力とする。

### D6: 測定スクリプトの実装方針
- `backend/scripts/measure_quality.py` 単一ファイル
- 既存 `app/services` を直接 import して再利用(API 経由ではない)
- 入力: job_id(`/api/process` 完了後の id)
- 出力: JSON + Markdown チェックリスト

**Alternatives**:
- API エンドポイントとして実装: 余計な依存、 batch 実行に不向き → ❌
- 別リポジトリ: メンテ分散、 まず単一リポで → ❌

### D7: 後続 change の運用ルール
ベースライン測定で不合格となった項目は **1 項目 = 1 change** で起こす(複数の修正をまとめない)。これにより:
- 修正が **退行を起こした場合に切り戻しやすい**
- 各 change の責任範囲が明確
- レビューしやすい

例外: 強く相互依存する項目(例: #1 ストーリー保全 と #2 冒頭末尾保護)は 1 change にまとめる判断もあり。

## Risks / Trade-offs

### R1: 目視判定の主観性
**Risk**: #1, #5, #9, #10 など目視判定の項目は判定者によって結果がブレる。
**Mitigation**:
- 期待値ドキュメントに **判定根拠** を具体的に書く(キーワードリスト、 想定 HOOK の方向性)
- ベースライン測定時の判定者と日付を記録 → 後で参照可能
- 長期的には Claude Vision 等で自動化(長期 backlog)

### R2: テスト動画 3 本では不十分
**Risk**: 3 本でカバーできないジャンル/シーンの動画でも品質ラインを満たしたい。
**Mitigation**:
- まず 3 本で運用、 月次レビューで追加候補を判断
- 業務量産で「うまくいかなかった動画」は順次 test-videos/ に追加する運用ルール

### R3: 機械測定のしきい値妥当性
**Risk**: 「動画長 60-120 秒」「文字数 8-14」などのしきい値はユーザーの体感で決めており、 実データで再評価が必要。
**Mitigation**: ベースライン測定で実値を確認 → 必要なら spec の MODIFIED Requirements で調整。

### R4: 不合格項目が多すぎる場合の運用
**Risk**: ベースラインで 14 項目中 8 個不合格、 など実装負荷が高い結果が出る可能性。
**Mitigation**:
- 優先順位を proposal に明示(業務量産直結 = ストーリー保全 > 字幕同期 > その他)
- 不合格項目を **1 change/1 項目** で順次潰す。 全部を 1 change に押し込まない。

### R5: 完成定義の硬直化
**Risk**: 14 項目を spec として確定すると、 「新しい品質次元」を追加しづらくなる。
**Mitigation**:
- 新項目の追加は spec の ADDED Requirements として後付け可能(spec-driven の流儀)
- 半年に 1 度、 品質基準を見直す運用ルール

## Migration Plan

### 段階 1: 本 change の完了
1. 14 項目 spec を確立
2. test-videos/ 整備
3. measure_quality.py 実装
4. ベースライン測定

### 段階 2: 不合格項目への対応
1. `baseline.md` で不合格になった項目を確認
2. 各項目について新規 change を `/openspec-propose`
3. 実装 → 再測定 → ベースラインを更新

### 段階 3: 業務量産投入
1. 14 項目すべて合格 or 許容範囲内
2. 業務動画(週 14 本)で運用開始
3. 失敗した動画を test-videos/ に追加 → 退行を防ぐ

### Rollback
本 change は **新規ファイル追加のみ** で既存実装に変更なし。 不要になったら `openspec/changes/establish-quality-line/` と `openspec/specs/quality-line/` と `test-videos/` と `backend/scripts/measure_quality.py` を削除すれば完全に元に戻る。

- **Q4**: baseline.md の永続化は change を archive した後どうするか?
  - archive で `openspec/changes/archive/establish-quality-line/baseline.md` に残るので参照可能
  - 最新の baseline は `openspec/specs/quality-line/baseline-latest.md` のようなパスに分離する選択肢もある
